The Core Question — Who Sends the Request?

Before anything else — SSRF vs CSRF comes down to one single question. Answer it correctly and the rest makes sense automatically.

Who is making the HTTP request?
🌐 The Browser
🖥️ The Server
🌐 Browser makes request
= CSRF

The attacker tricks your browser into sending a request. The request comes from your machine, your session, your cookies.

🖥️ Server makes request
= SSRF

The attacker tricks the server itself into making a request. The request comes from the server — giving access to things only the server can reach.

Why does it matter where the request comes from?

A server lives inside the network. It can talk to databases, admin panels, internal APIs, and cloud metadata endpoints that your browser can never reach directly. When the server is tricked into making a request, it carries its own trust and network access with it.

02

CSRF vs SSRF — Side by Side

CSRF — Cross-Site Request Forgery
  • Attacker tricks your browser to make a request
  • Request comes from the client side
  • Uses your existing login session
  • Target: things your user account can do
  • Example: Transfer money on your behalf
SSRF — Server-Side Request Forgery
  • Attacker tricks the server to make a request
  • Request comes from the server side
  • Uses the server's internal network access
  • Target: internal systems the server can reach
  • Example: Access admin panel at 127.0.0.1

Key insight:

In SSRF, the attacker never directly touches the internal system. They use the server as a proxy. The server thinks it's doing a normal job (fetching a URL). It doesn't realize it's being steered into restricted territory.

03

What is SSRF?

"Make the server fetch a URL you control — and steer it somewhere it shouldn't go."

SSRF (Server-Side Request Forgery) is a vulnerability where an attacker can cause the server-side application to make HTTP requests to an arbitrary domain or IP — including internal services the attacker has no direct access to.

Think of it like this: the web server has keys to rooms you don't. SSRF lets you borrow those keys by giving the server bad directions.

Filter Bypass Techniques

Developers try to block SSRF by checking if the URL contains "127.0.0.1" or "localhost". Attackers bypass this by encoding the same address in different ways. The OS resolves them all to the same IP.

The problem with naive blocklists

If you only block the string 127.0.0.1, an attacker can use dozens of equivalent representations that all resolve to the same address. The server's network stack doesn't care how you write the IP — it ends up at the same place.

How to Prevent SSRF

❌ VULNERABLE PATTERN
url = request.args.get('url') resp = requests.get(url) # No validation whatsoever # Attacker controls destination
✅ SECURE PATTERN
ALLOWED = ['cdn.trusted.com','api.example.com'] parsed = urlparse(url) if parsed.hostname not in ALLOWED: raise ValueError("Blocked") resp = requests.get(url)
The most important defense: Network-level controls

Even with perfect URL validation in code, a zero-day or misconfiguration can bypass it. The most reliable protection is ensuring the web server cannot reach internal systems at the network level — egress filtering, service mesh policies, or hosting internal services on separate VLANs with no route from the DMZ.
